# 跨域协作操作手册

## 核心原则

**ID 是跨域查询的钥匙。** 当一个工具的返回结果中包含其他域的 ID（如
productId、userId、orderId），并且用户需要该域的详细信息时，必须主动调用对应工具补查，不能复用上一步已有的摘要信息。

### 各工具返回的跨域 ID 一览

| 来源工具                         | 返回结果中包含的跨域 ID                           | 说明                              |
|------------------------------|-----------------------------------------|---------------------------------|
| order_tool_agent（订单详情）       | productId（商品列表中）、userId（用户信息中）          | 商品只有 ID/名称/数量/单价，用户只有 ID/昵称/手机号 |
| after_sale_tool_agent（售后详情）  | orderId、orderNo、userId、productId（商品信息中） | 商品信息同样是摘要，用户只有 ID 和昵称           |
| analytics_tool_agent（排行/退货率） | 商品相关的 ID                                | 只有统计汇总，不含商品经营和药品专业信息            |
| user_tool_agent（用户详情）        | 无直接跨域 ID，但用户信息可间接用于订单/售后查询              | 需要用用户信息作为筛选条件                   |

### 数据依赖链（极其重要！）

```
user_tool_agent —— 需要 userId 才能查 —— userId 只能从 order/after_sale 获得
product_tool_agent —— 需要 productId 才能查 —— productId 只能从 order/after_sale/analytics 获得
order_tool_agent —— 可以直接查，也可用 orderNo/orderId 从 after_sale 获得
```

**绝对禁止的调用方式**：

- 直接调 user_tool_agent 说"查最新订单的用户" → 它没有订单数据！
- 直接调 product_tool_agent 说"查订单里的商品" → 它没有订单数据！
- 调完上游后调下游说"查这些商品的详情" → "这些"是模糊描述，必须传具体 ID！
- task_description 写 `"查询用户 ID 为 [从上一步获取的userId] 的详情"` → 子代理是无状态的，它会把 `[从上一步获取的userId]`
  当作字面文本！
- task_description 写 `"查询上一步返回的商品详情"` → 子代理看不到"上一步"，它没有任何上下文！

** 子代理完全无状态！** 每个子代理是独立的 AI，无法访问对话历史和其他子代理的返回结果。task_description 是它的**唯一输入**
。你必须：

1. **等待上一步子代理实际返回数据**
2. **从返回结果中提取真实的 ID 数值**（如 userId=1001, productId=2001）
3. **将真实数值直接写入下一步的 task_description**

绝不能在上一步还未返回时就用占位符 `[xxx]`、`{xxx}`、`<xxx>` 预构建 task_description。

### 摘要 vs 详情对照

**订单/售后中的商品信息（摘要）只有 6 个字段**：

- 商品 ID、名称、图片、单价、数量、总价

**商品详情额外提供**：

- 库存、分类名称、上下架状态、配送方式、商品图片列表等

**药品详情额外提供**：

- 功效/适应症、用法用量、不良反应、禁忌、注意事项、成分、说明书全文、生产单位、批准文号、是否处方药等

## 场景 1：查订单中的商品详细信息

> **用户问法**："帮我查最新 10 个订单里每个商品的详细信息"、"这些订单的商品库存还有多少？"、"
> 订单里面买的都是什么分类的商品？"

**操作步骤**：

1. 调用 `order_tool_agent`
    - task_description：`"查询最新 10 个订单的详情，我需要每个订单中所有商品的商品 ID（productId）列表"`
    - 必须明确要求返回 productId，否则后续无法调商品工具
2. 从返回结果中收集所有不重复的 **商品 ID**
3. 调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001、2003、2005、2008 的商品详情，包括价格、库存、分类和上下架状态"`
    - 必须传入具体的商品 ID 数值，不能说"查订单里的商品"
4. 将订单中的购买记录与商品详细信息合并呈现

**关键关注**：第 3 步返回的库存数量、分类名称、上下架状态是订单中没有的。

## 场景 2：查订单中药品的禁忌和适应症

> **用户问法**："这些订单里的药有什么禁忌？"、"订单里买的药说明书帮我查一下"、"这些商品有没有处方药？"

**操作步骤**：

1. 调用 `order_tool_agent`
    - task_description：`"查询订单 O20260101 的详情，我需要订单中所有商品的商品 ID（productId）列表"`
2. 从商品列表中提取 **商品 ID**
3. 调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001、2003 的药品详情，包括功效/适应症、用法用量、不良反应、禁忌、注意事项和是否处方药"`
    - 要药品专业信息必须说"药品详情"而不是"商品详情"
4. 按商品逐一展示药品信息

**关键关注**：药品的功效、禁忌、用法、不良反应、是否处方药都只有药品详情工具能提供。

## 场景 3：查订单买家的详细资料

> **用户问法**："下单的用户是谁？详细信息帮我看一下"、"这个订单的买家资料"、"订单的用户什么时候注册的？"

**操作步骤**：

1. 调用 `order_tool_agent`
    - task_description：`"查询订单 O20260101 的详情，我需要下单用户的用户 ID（userId）"`
    - 明确要求返回 userId！不能指望 user_tool_agent 能根据订单号查用户
2. 从用户信息中提取 **用户 ID**
3. 调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的详情，包括注册时间、最后登录时间、手机号、邮箱和消费统计"`
    - 必须传入具体的用户 ID 数值
4. 合并订单中的下单信息和用户详细资料一起汇报

**关键关注**：订单中的用户信息只有 ID、昵称、手机号，不含邮箱、身份证、注册时间、登录记录、消费统计等。

**绝对错误的做法**：直接调 `user_tool_agent("查询订单 O20260101 的用户信息")` → user_tool_agent 没有订单数据，无法执行！

**操作步骤**：

1. 调用 `order_tool_agent`
    - task_description：`"查询订单 O20260101 的详情，我需要下单用户的用户 ID（userId）"`
2. 调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的钱包信息，包括余额和钱包状态"`
3. （可选）调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的消费信息"`
4. 汇总钱包和消费情况

## 场景 5：售后对应的订单详情

> **用户问法**："这个售后对应的订单什么情况？"、"售后单关联的订单详情看一下"、"售后订单什么时候发的货？"

**操作步骤**：

1. 调用 `after_sale_tool_agent`
    - task_description：`"查询售后 ID 为 30001 的详情，我需要关联的订单 ID（orderId）和订单编号（orderNo）"`
2. 从结果中提取 **订单 ID 或订单编号**
3. 调用 `order_tool_agent`
    - task_description：`"查询订单 ID 为 10001 的详情，包括收货地址、支付信息和商品明细"`
4. （可选）如需物流信息，继续调用 `order_tool_agent`
    - task_description：`"查询订单 ID 为 10001 的发货记录"`
5. 将售后信息和订单信息整合呈现

## 场景 6：售后商品的完整信息和说明书

> **用户问法**："售后涉及的商品详细信息看一下"、"退货的这个药说明书帮我查查"、"售后商品的库存和价格"

**操作步骤**：

1. 调用 `after_sale_tool_agent`
    - task_description：`"查询售后 ID 为 30001 的详情，我需要涉及商品的商品 ID（productId）"`
2. 调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001 的商品详情，包括价格、库存、分类和上下架状态"`
3. （如需药品信息）调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001 的药品详情，包括说明书、功效、禁忌和用法用量"`
4. 合并售后信息 + 商品信息 + 药品信息

## 场景 7：售后用户的账号状态和消费记录

> **用户问法**："提售后的这个人是什么情况？"、"是不是新注册的号？消费多少？"

**操作步骤**：

1. 调用 `after_sale_tool_agent`
    - task_description：`"查询售后 ID 为 30001 的详情，我需要申请用户的用户 ID（userId）"`
    - 必须先从售后中拿到 userId，不能直接让 user_tool_agent 查"售后的用户"
2. 调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的详情，包括注册时间、总订单数、总消费金额和账号状态"`
3. （可选）调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的消费信息"`
4. 综合判断用户资质并汇报

**绝对错误的做法**：直接调 `user_tool_agent("查询售后 30001 的用户信息")` → user_tool_agent 没有售后数据！

## 场景 8：用户买过什么药、这些药有什么注意事项

> **用户问法**："用户 1001 买过什么药？这些药有什么注意事项？"、"看看他买的药有没有禁忌冲突"

**操作步骤**：

1. 调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的消费信息"`
2. 从消费记录中获取关联订单信息，调用 `order_tool_agent`
    - task_description：`"查询订单 ID 为 10001、10003、10005 的详情，我需要所有商品的商品 ID（productId）列表"`
    - 必须传入从消费记录中提取的具体订单 ID
3. 收集所有不重复的商品 ID，调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001、2003、2008 的药品详情，包括功效、用法用量、不良反应、禁忌和注意事项"`
4. 按药品逐一汇总注意事项

**关键关注**：这是一个三步跨域链路（用户 → 订单 → 药品），每一步都是为了获取下一步需要的 ID。任何一步的 task_description
不够明确都会导致链路断裂。

3. 调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001、2003、2005、2008、2010 的商品详情，重点关注库存数量和上下架状态"`
4. 对比销量和库存，给出库存是否充足的分析

## 场景 11：完整排查一笔售后（全链路）

> **用户问法**："帮我全面排查一下售后 30001，包括订单、用户、商品所有信息"

**操作步骤**：

1. 调用 `after_sale_tool_agent`
    - task_description：`"查询售后 ID 为 30001 的详情，我需要关联的订单 ID（orderId）、用户 ID（userId）和商品 ID（productId）"`
    - 一次性要求所有跨域 ID，避免后续还需再调一次
2. 调用 `order_tool_agent`
    - task_description：`"查询订单 ID 为 10001 的详情，包括收货地址、支付信息和发货记录"`
3. 调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的详情，包括注册时间、消费统计和账号状态"`
4. 调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001 的商品详情，包括价格、库存和分类"`
5. 调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001 的药品详情，包括说明书、功效、禁忌和用法用量"`
6. 汇总五部分信息，形成完整的售后排查报告

**关键关注**：这是最完整的全链路查询，覆盖售后→订单→用户→商品→药品全部维度。第 1 步的 task_description 质量直接决定后续 4
步是否能顺利执行。

## 场景 15：查最新订单的商品详细信息和用户信息（多下游并行）

> **用户问法**："我需要最新订单的商品详细信息和这个订单的用户信息"

**操作步骤**：

1. 调用 `order_tool_agent`
    - task_description：`"查询最新订单的详情，我需要订单中所有商品的商品 ID（productId）列表以及下单用户的用户 ID（userId）"`
    - 这是关键！因为后续需要同时查商品和用户，必须在第一步一次性要求两个 ID
2. 从返回中提取：productId = [2001, 2003]，userId = 1001
3. 调用 `product_tool_agent`
    - task_description：`"查询商品 ID 为 2001、2003 的商品详情，包括价格、库存、分类和上下架状态"`
4. 调用 `user_tool_agent`
    - task_description：`"查询用户 ID 为 1001 的用户详情，包括注册时间、手机号、消费统计"`
5. 合并订单信息 + 商品详细信息 + 用户信息

**错误做法**：

- 第一步只要了 productId 没要 userId → 查完商品后发现还要查用户，却没有 userId，只能再调一次 order_tool_agent
- 直接调 user_tool_agent 说"查最新订单的用户" → user_tool_agent 没有订单数据，无法执行

## 通用操作原则

1. **先判断是否需要跨域**：用户只要问到"详细""完整""更多"或者明确提及另一个域的信息，就需要跨域补查
2. **预判所有下游 ID 需求**：在调用第一个子代理时，分析用户问题需要哪些下游工具，一次性要求返回所有需要的跨域 ID
3. **传 ID 要准确明确**：调用下游工具时，在自然语言描述中写清具体 ID 值（如"查询商品 ID 为 2001、2003 的详情"），严禁使用"
   这些""那些"等模糊指代
4. **理解数据归属边界**：user_tool_agent 和 product_tool_agent 是终端节点，它们不持有订单/售后数据，必须由上游工具提供具体
   ID 才能查询
5. **去重后批量查询**：多个订单可能包含相同商品，收集商品 ID 时先去重，避免重复查询
6. **信息合并时标注来源**：汇总时按订单/商品/用户等维度清晰组织，用表格或列表呈现
7. **不要复用旧数据**：当上下文中已有某些数据但用户要求"详细信息"时，仍需重新调用工具获取最新完整数据

## task_description 质量检查清单

在写出 task_description 之前，快速过一遍这个清单：

- [ ] **是否包含占位符？** → 检查有没有 `[xxx]`、`{xxx}`、`<xxx>`、`从上一步获取的`、`上一步返回的` 等写法 → 如果有，*
  *立即替换为真实数值**
- [ ] **上一步是否已经返回了？** → 如果还没返回，不能预构建 task_description，必须等返回后再提取 ID
- [ ] 是否明确写了"我需要 xxx ID"？（当后续需要跨域调用时）
- [ ] 是否传入了具体的 ID 值？（调用下游工具时，如 `1001`、`2001` 这样的真实数字）
- [ ] 是否向正确的子代理请求数据？（没有向 user_tool_agent 要订单数据？）
- [ ] task_description 中的关注点是否与用户问题匹配？（用户问库存就要关注库存，问禁忌就要关注药品详情）
- [ ] 多个下游需求是否在第一步一次性要求了所有 ID？
