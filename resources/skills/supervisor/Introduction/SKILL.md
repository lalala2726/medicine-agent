# Supervisor 子工具调用指南

## 技能目标

帮助 supervisor_agent 正确使用 5 个子代理工具，重点解决两个问题：

1. **单工具使用**：每个子代理能做什么、参数怎么传、返回哪些数据、**不返回**哪些数据
2. **跨域协作**：一个工具的返回结果包含其他域的 ID 时，如何接力调用下一个工具获取完整信息

## 子代理工具能力总览

| 子代理                   | 内部工具                                   | 核心能力                      |
|-----------------------|----------------------------------------|---------------------------|
| order_tool_agent      | 订单列表 / 订单详情 / 流程时间线 / 发货记录             | 订单筛选浏览、订单完整信息查询、状态追踪、物流履约 |
| product_tool_agent    | 商品列表 / 商品详情 / 药品详情                     | 商品搜索筛选、商品基础信息、药品说明书与专业信息  |
| user_tool_agent       | 用户列表 / 用户详情 / 钱包 / 钱包流水 / 消费信息         | 用户搜索筛选、用户资料核验、资金状态与消费行为排查 |
| after_sale_tool_agent | 售后列表 / 售后详情                            | 售后申请筛选、处理进度追踪、售后全链路信息     |
| analytics_tool_agent  | 运营总览 / 订单趋势 / 状态分布 / 支付分布 / 热销排行 / 退货率 | 经营数据总览、趋势分析、排行与分布统计       |

## 跨域协作指南（核心）

当用户的需求**跨越多个业务域**时（如"查订单里的商品详细信息"），需要多工具接力调用：

- [跨域协作操作手册](reference/cross_agent.md) — 12 种常见跨域场景的完整分步操作指引

### 核心原则速记

1. **ID 是跨域的钥匙**：拿到 productId → 调商品工具；拿到 userId → 调用户工具；拿到 orderId → 调订单工具
2. **摘要 ≠ 详情**：订单/售后中的商品信息只有名称、数量、单价，不含库存/分类/说明书
3. **用户要"详细"就要补查**：听到"详细信息""完整信息""更多信息"等关键词，必须跨域获取
4. **传 ID 要明确**：调下游工具时，在任务描述中写清具体的 ID 值

## task_description 构建规范（关键能力）

子代理只接受一个 `task_description` 参数（自然语言），你写的描述质量直接决定子代理能否返回正确、完整的数据。

### 最高优先级：子代理完全无状态！

每个子代理是独立的无状态 AI，**无法访问对话上下文、无法看到其他子代理的返回结果**。
task_description 是子代理的**唯一输入**，必须包含所有执行所需的具体信息。

**严禁使用任何占位符语法**：`[从上一步获取的userId]`、`[订单中的productId]`、`{userId}`、`上一步返回的xxx`
等写法，子代理会把它们当作字面文本，导致查询失败。

**正确做法**：等待上一步返回 → 提取真实 ID 数值 → 将真实数值写入 task_description。

### 核心规则

| 规则             | 说明                                                                                |
|----------------|-----------------------------------------------------------------------------------|
| **预判下游 ID 需求** | 在调用第一个子代理时，预判后续步骤需要哪些 ID（userId / productId / orderId），在 task_description 中明确要求返回 |
| **指定具体字段**     | 不说"查一下详情"，要说"我需要 xxx、xxx 字段信息"                                                    |
| **传入具体 ID 值**  | 调下游子代理时，必须写入从上一步提取的**真实具体 ID 数值**，不能写"这些""那个"，更不能写占位符如 `[userId]`                 |
| **理解数据归属**     | 不要向 user_tool_agent 要订单数据，不要向 product_tool_agent 要订单数据 —— 它们没有！                   |

### 数据归属关系

```
order_tool_agent 提供 → userId, productId  （可以作为上游）
after_sale_tool_agent 提供 → orderId, userId, productId（可以作为上游）
analytics_tool_agent 提供 → productId（排行中）（可以作为上游）
user_tool_agent → 终端节点，只接受 userId，无法根据订单/售后查用户
product_tool_agent → 终端节点，只接受 productId/名称，无法根据订单查商品
```

### 常见错误

| 错误做法                                    | 💥 后果                                | 正确做法                                                         |
|-----------------------------------------|--------------------------------------|--------------------------------------------------------------|
| 调 order_tool_agent 只说"查最新订单"，后续又要查用户和商品 | 返回中可能未突出 userId/productId，导致提取困难或遗漏  | 第一步就说"查最新订单详情，需要用户 ID 和所有商品 ID"                              |
| 直接调 user_tool_agent 说"查最新订单的用户信息"       | user_tool_agent 没有订单数据，完全无法执行        | 先调 order_tool_agent 拿 userId，再调 user_tool_agent              |
| 调 product_tool_agent 说"查订单里的商品详情"       | product_tool_agent 没有订单数据，不知道订单有哪些商品 | 先调 order_tool_agent 拿 productId，再传具体 ID 给 product_tool_agent |
| 拿到 ID 后调子代理说"查这些商品的详情"                  | "这些"是模糊描述，子代理不知道具体 ID                | 写"查询商品 ID 为 2001、2003、2005 的详情"                              |
| task_description 写 `[从上一步获取的userId]`    | 子代理无状态，会把中括号内容当字面文本                  | 等上一步返回后提取真实 ID，写实际数值如 `1001`                                 |
| task_description 写"查询上一步返回的商品详情"        | 子代理看不到"上一步"，没有上下文                    | 写"查询商品 ID 为 2001、2003 的商品详情"                                 |
| 一次 task_description 里塞太多不相关请求           | 子代理可能只处理部分请求                         | 将不同类型的请求拆分为多次调用                                              |
